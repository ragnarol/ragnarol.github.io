# torch.md — Handoff guide for this repository

Read this before touching anything. It explains what this repo is, how it is
built, the conventions everything depends on, and the traps that are not
obvious from looking at the file tree.

## What this is

A **GitHub Pages site** (Jekyll) that publishes **bilingual (Spanish/English)
session summaries of tabletop RPG campaigns**. It is served at
**www.juancasanova.dev** (see `CNAME`) from the `master` branch of
`ragnarol/ragnarol.github.io`.

There is no application code, no build pipeline in CI, no tests. GitHub Pages
builds the site automatically with Jekyll on every push to `master`. The
"work" in this repo is almost entirely adding Markdown content and
occasionally tweaking the one layout file or the CSS.

Current campaigns:

| Campaign | Directory | Sessions | Theme |
|---|---|---|---|
| Pirates of Drinax (Traveller) | `es/pirates-of-drinax/`, `en/pirates-of-drinax/` | 01–67 | `sci-fi` (dark blue, monospace) |
| Ravenloft / Curse of Strahd (D&D) | `es/ravenloft/`, `en/ravenloft/` | 01–13 (with `.5` interludes) | `gothic` (dark red, serif) |

## Architecture

```
_config.yml            Jekyll config. Assigns layout/lang/theme by path (see below)
_layouts/default.html  THE one layout. Sidebar, language toggle, title logic, giscus comments
assets/style.css       Base stylesheet (GitHub-like, light)
assets/themes/*.css    Per-campaign overrides (sci-fi.css, gothic.css), loaded on top of style.css
index.md               Root landing page (language chooser)
es/index.md            Spanish home        en/index.md   English home
es/<campaign>/*.md     Spanish summaries   en/<campaign>/*.md  English summaries
CNAME                  Custom domain (www.juancasanova.dev) — do not delete or edit
build_site.sh          LEGACY. Do not run. See "Traps" below
```

### Everything is driven by directory conventions, not frontmatter

Content files have **no YAML frontmatter** (only the three `index.md` files
do). Instead, `_config.yml` uses `defaults` scoped by path to inject values:

- everything gets `layout: default`
- anything under `es/` gets `lang: es`; under `en/` gets `lang: en`
- anything under `*/pirates-of-drinax` gets `theme: sci-fi`; under
  `*/ravenloft` gets `theme: gothic`

The layout then reads `page.theme` to add
`assets/themes/<theme>.css` after the base stylesheet, and reads `page.lang`
to build the sidebar and the language toggle.

**Consequence:** to add a session you just drop a `.md` file in the right
directory. No frontmatter, no config change, no registration anywhere.

### The layout derives everything from the filename and path

`_layouts/default.html` is the only template and does three clever things:

1. **Page title (both `<title>` and the visible `<h1>`)** is the filename with
   `.md`, `- eng`, `- summary`, dashes and underscores stripped/replaced. The
   `# Heading` inside the Markdown file is *also* rendered, so pages show the
   derived filename title followed by the file's own heading. Filenames are
   therefore user-facing text — name them like titles.
2. **Sidebar navigation** is generated by splitting each page's `path` on `/`:
   part `[1]` is the campaign, and pages are grouped into a `<details>`
   accordion per campaign (open for the campaign you're currently reading).
   Pages are sorted by `path`, so the **numeric filename prefix controls the
   order** (`01`, `04.5`, `13`, `67`…).
3. **Comments** via [giscus](https://giscus.app) (GitHub Discussions), with the
   repo ID hardcoded in the layout. Add `comments: false` frontmatter to a
   page to suppress them.

### File naming convention

```
NN[.5] - <Title>.md                      e.g. "06.5 - Arañas de Castillo.md"
NN - PoD - <Arc> NN - summary.md         older Spanish Drinax files
NN - PoD - <Arc> NN - eng.md             English Drinax files
```

- `NN` is a zero-padded session number; `.5` marks interludes/side sessions
  (Ravenloft).
- The `- summary` / `- eng` suffixes are **cosmetic leftovers** — the layout
  strips them for display. Newer files (Drinax ≥ 61) dropped `- summary`
  entirely. When adding new files, matching the recent style (no suffix on
  Spanish, `- eng` on English) is fine; just keep the number prefix.
- Filenames contain spaces and accents on purpose; the URL is derived from the
  path and it works. Keep the Spanish and English filenames' *numbers* in
  sync so sessions pair up.

### Content format of a session summary

Each summary follows a consistent structure (keep it when generating new
ones):

```markdown
# Session title

**PCs present:** ...
**NPCs present:** ...

### Key Developments        (### Desarrollos Principales)
### Loose Ends              (### Cabos Sueltos)
### Session Joke            (### Broma de la Sesión)

---

## Escena 1: ...            (scene-by-scene breakdown with PCs/NPCs/places)
```

The Spanish file is the original (summaries are written from Spanish session
transcripts); the English file is a translation of it. Note that kramdown is
configured with `hard_wrap: true`, so **single newlines become `<br>`** —
don't hard-wrap prose lines in the Markdown.

## How to do common tasks

### Add a new session summary

1. Create `es/<campaign>/NN - <Título>.md` with the summary (no frontmatter).
2. Create the matching `en/<campaign>/NN - <Title> - eng.md` translation.
3. Commit and push to `master`. GitHub Pages rebuilds automatically; there is
   nothing else to do.

### Add a new campaign

1. Create `es/<campaign-slug>/` and `en/<campaign-slug>/` and add session
   files. The sidebar picks up the new campaign automatically from the path
   (displayed name = slug with dashes → spaces, capitalized).
2. (Optional but expected) create `assets/themes/<name>.css` following the
   pattern of `sci-fi.css` / `gothic.css` (override the `:root` CSS variables
   from `style.css`, restyle `.sidebar`, `.content`, headings), then add two
   scope blocks in `_config.yml` mapping `es/<campaign-slug>` and
   `en/<campaign-slug>` to `theme: <name>`.

### Preview locally (optional)

There is no Gemfile. Quickest check:

```bash
gem install jekyll kramdown
jekyll serve   # from the repo root; serves on http://localhost:4000
```

A full local build is rarely needed — pushes to `master` go live within a
minute or two, and mistakes are cheap to fix forward.

## Traps and gotchas (read this twice)

1. **Do NOT run `build_site.sh`.** It is a legacy bootstrap script from when
   the site was Drinax-only. Running it will **overwrite** `_config.yml`,
   `assets/style.css`, `_layouts/default.html`, and all three `index.md` files
   with old versions that have no themes, no campaign accordion, and the old
   "Piratas de Drinax" title. It also expects a `source/` directory that is
   gitignored. Treat it as historical reference only (or delete it if the
   owner agrees).
2. **`source/` is gitignored.** Raw session transcripts live only on the
   owner's machine. Never expect them in a fresh clone, and never commit them.
3. **Known content gap:** session **58 – "Yellow Hawk Derribado"** exists in
   Spanish (`es/pirates-of-drinax/`) but has **no English translation** in
   `en/pirates-of-drinax/`. Some English Ravenloft files also keep Spanish
   titles (`08 - Prenupcia`, `10 - Ataudes`, `13 - Me lo dijo una gitana
   loca`) — that's tolerated, not a bug to "fix" without asking.
4. **Filename typos matter** because filenames are displayed as titles and
   control sort order. Existing example: `63 -PoD - De compras.md` (missing
   space). If you rename a file, its URL changes — giscus comment threads are
   mapped by pathname, so renaming a page orphans its comments.
5. **A stray first line can leak into the page.** At least one file
   (`es/ravenloft/13 - ...`) begins with a leftover LLM preamble line ("Aquí
   tienes el resumen…") before the `#` heading. When generating summaries
   with an LLM, strip any such preamble before committing.
6. **Don't add frontmatter to content files** unless you specifically need to
   override a default (`comments: false` is the only sanctioned use). Layout,
   language and theme all come from `_config.yml` path scopes.
7. **The layout is load-bearing and fragile.** All display-name logic is
   chained Liquid `replace` filters in *two places* (the `<title>` and the
   `<h1>`/sidebar) — if you change the suffix-stripping rules, change them
   everywhere. Test by pushing to a branch and checking a rendered page, or
   with a local `jekyll serve`.
8. **giscus IDs are hardcoded** in `_layouts/default.html` (`data-repo-id`,
   `data-category-id`). They are tied to this exact repo's Discussions; if the
   site is ever forked/moved they must be regenerated at giscus.app.

## Mental model in one paragraph

This is a static content site where the *filesystem is the database*:
`<lang>/<campaign>/<number> - <title>.md`. One Jekyll layout turns that tree
into a themed, bilingual site with an auto-generated sidebar. To contribute,
you add correctly-named Markdown files in pairs (es + en) and push to
`master`. Avoid the legacy build script, keep filenames clean, and you cannot
really break anything.
